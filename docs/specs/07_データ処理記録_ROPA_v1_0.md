### 3.1 処理の目的と法的根拠の対応表

| **PA-004** | サービス改善 | (a) 同意 | ユーザーの明示的同意に基づく |
| **PA-005** | 課金管理 | (b) 契約の履行 | サブスクリプション管理に必要 |
| **PA-006** | 不正利用防止 | (f) 正当な利益 | サービスの安全性確保 |
| **PA-007** | 法的義務の遵守 | (c) 法的義務 | 法令遵守のため |
| **PA-008** | 将来のML学習 | (a) 同意 | ユーザーの明示的同意に基づく |

**参照**: プライバシーポリシーv3.1第5条(情報の利用目的)

### 3.2 正当な利益(Legitimate Interests)の詳細

**PA-006: 不正利用防止**の法的根拠として「正当な利益」を使用する理由:

#### 3.2.1 追求する利益

1. **サービスの安全性確保**: 不正アクセスやアカウント乗っ取りからユーザーを保護
2. **他のユーザーの保護**: 不正利用による他ユーザーへの影響を防止
3. **サービス品質の維持**: システムの安定稼働を確保

#### 3.2.2 必要性の評価

- 不正利用の防止は、サービスの安全性維持に不可欠
- 代替手段では同等の保護を実現できない
- 最小限のデータ(アクセスログ、デバイス情報)のみを使用

#### 3.2.3 利益衡量テスト(Balancing Test)

| 評価項目 | 管理者の利益 | データ主体の権利 | 結果 |
|---------|------------|----------------|------|
| **処理の目的** | サービス全体の安全性確保 | プライバシーへの軽微な影響 | ✅ 管理者の利益が優先 |
| **データの性質** | 非機微な技術的データ | 個人を直接特定しない | ✅ 侵害性が低い |
| **データの範囲** | 必要最小限 | 過度な収集なし | ✅ データ最小化原則を遵守 |
| **ユーザーの期待** | セキュリティ対策を期待 | 一般的な予測可能性 | ✅ 合理的な期待の範囲内 |
| **代替手段** | 他の手段では不十分 | オプトアウト可能 | ✅ 必要性が高い |

**結論**: 正当な利益に基づく処理は適法である。

**参照**: プライバシーポリシーv3.1第5.6条

| **生体認証データ** | ❌ 収集しない | 骨格座標は「参考情報」として使用され、個人の一意な識別には使用しない |
| **人種・民族データ** | ❌ 収集しない | - |
| **政治的見解** | ❌ 収集しない | - |
| **宗教的信条** | ❌ 収集しない | - |
| **労働組合への加入** | ❌ 収集しない | - |
| **性生活・性的指向** | ❌ 収集しない | - |
| **遺伝データ** | ❌ 収集しない | - |

**重要な位置づけ**(要件定義書v3.1第1.2.2項、プライバシーポリシーv3.1第2.3条):

- 骨格座標データは「参考情報」として使用されます

- 最終的な判断は利用者自身の責任において行うものです

## 8. 技術的・組織的セキュリティ対策

### 8.1 技術的セキュリティ対策

#### 8.1.1 暗号化

| 暗号化対象 | 暗号化方式 | 鍵管理 | 実装 |
|----------|----------|-------|------|
| **通信** | TLS 1.3 | 自動管理 | Firebase、GCP |
| **データベース** | AES-256 | Cloud KMS | Firestore、BigQuery |
| **ローカルDB** | AES-256 | デバイスキーチェーン | Hive |
| **パスワード** | bcrypt | - | Firebase Authentication |
| **ユーザーID** | SHA-256(仮名化) | - | BigQueryへの転送時 |

**参照**: プライバシーポリシーv3.1第8.4条、システムアーキテクチャ設計書v3.1第8章

#### 8.1.2 アクセス制御

| 制御レベル | 実装方法 | 対象 |
|----------|---------|------|
| **ユーザー認証** | Firebase Authentication | 全ユーザー |
| **データベースアクセス** | Firebase Security Rules | Firestore |
| **GCPリソースアクセス** | Cloud IAM | BigQuery、Cloud Functions |
| **APIアクセス** | JWT認証 | Cloud Functions |
| **管理者アクセス** | MFA必須 | 管理コンソール |

**Firebase Security Rulesの例**:
```javascript
rules_version = '2';
service cloud.firestore {
  match /databases/{database}/documents {
    // ユーザーは自分のデータのみアクセス可能
    match /users/{userId} {
      allow read, write: if request.auth != null && request.auth.uid == userId;
    }
    
    // トレーニングデータも同様
    match /training_sessions/{sessionId} {
      allow read, write: if request.auth != null && 
                            resource.data.userId == request.auth.uid;
    }
  }
}
```

**参照**: システムアーキテクチャ設計書v3.1第8.2節

#### 8.1.3 データ最小化とプライバシーバイデザイン

| 原則 | 実装 |
|-----|------|
| **オンデバイス処理** | カメラ映像は端末上でのみ処理、サーバーに送信しない |
| **仮名化** | BigQueryへの転送時、ユーザーIDをSHA-256でハッシュ化 |
| **データ最小化** | 必要最小限のデータのみ収集 |
| **自動削除** | アカウント削除時に全データを自動削除 |

**参照**: プライバシーポリシーv3.1第8.1条、第8.3条

### 8.2 組織的セキュリティ対策

#### 8.2.1 アクセス管理

| 役割 | アクセス権限 | 対象リソース |

### 12.2 ROPAの更新トリガー

以下のいずれかが発生した場合、ROPAを更新します:

1. **新しい処理活動の開始**: 新機能の追加
2. **処理目的の変更**: 既存機能の目的変更
3. **新しい受領者の追加**: 第三者サービスの追加
4. **第三国転送の開始**: 新しい国へのデータ転送
5. **データカテゴリーの追加**: 新しい種類のデータ収集
6. **法令の変更**: GDPRや関連法令の改正
7. **重大なセキュリティインシデント**: データ侵害の発生

## 付録C: 用語集

| 用語 | 定義 |
|-----|------|
| **ROPA** | Record of Processing Activities(処理活動の記録) - GDPR第30条 |
| **管理者(Controller)** | 個人データの処理の目的と手段を決定する者 |
| **処理者(Processor)** | 管理者に代わって個人データを処理する者 |
| **データ主体(Data Subject)** | 個人データによって識別される自然人 |
| **DPO** | Data Protection Officer(データ保護責任者) - GDPR第37条 |
| **DPA** | Data Processing Agreement(データ処理契約) - GDPR第28条 |
| **SCC** | Standard Contractual Clauses(標準契約条項) |
| **EU-US DPF** | EU-US Data Privacy Framework(EU-米国データプライバシーフレームワーク) |
| **仮名化** | 追加情報なしでは特定の個人に結びつけられないようにすること - GDPR第4条5項 |
| **匿名化** | 個人の特定ができないようにすること |
| **正当な利益** | 管理者または第三者の正当な利益 - GDPR第6条1項(f) |
| **利益衡量テスト** | 正当な利益とデータ主体の権利のバランスを評価するプロセス |
